# Trading Orders & Positions Production Spec

## Goal

The trading terminal must behave like a live paper-trading module: every order action has one source of truth, visible order status feedback, wallet movement, and positions that refresh together.

## End-to-End Flow

1. User selects a market and strike from the option chain.
2. Ticket derives the active executable quote:
   - BUY routes against ask.
   - SELL routes against bid.
3. User chooses order mode:
   - MARKET executes immediately at the active quote if liquidity exists.
   - LIMIT executes immediately only when marketable; otherwise it rests as a working order.
4. Frontend sends `POST /api/v1/orders` with:
   - `matchId`, `marketId`, `strike`, `side`, `type`, `quantity`, `price`.
5. Backend validates:
   - user auth, live match, tradable market, strike exists, quantity positive, price positive for LIMIT, wallet capacity for BUY, open position capacity for SELL.
6. Backend creates the order with `status=open`.
7. Backend reserves BUY margin.
8. If executable, backend creates an internal execution ledger entry, settles wallet, and updates the order to:
   - `executed` when the full quantity is done.
   - `partially_filled` when only part of the quantity is done.
9. Frontend receives the created/final order response and immediately updates the order cache.
10. Frontend refetches active terminal queries:
    - orders, open positions, wallet, portfolio.
11. UI displays:
    - Orders tab: valid order history and live working orders.
    - Positions tab: derived open positions.
    - Wallet pill and ticket cash values from wallet API.

## Data Contract

Order backend status values:

- `open` -> frontend `PENDING`
- `partially_filled` -> frontend `PARTIAL`
- `executed` -> frontend `FILLED`
- `cancelled` -> frontend `CANCELLED`
- `rejected` -> frontend `REJECTED`
- any unknown status -> frontend `UNKNOWN`

Working orders must satisfy all of:

- status is `PENDING` or `PARTIAL`
- `remainingQuantity > 0`
- `strike > 0`

Malformed legacy rows must not be shown as cancellable working orders.

## Production Risk Scan

Resolved in current UI pass:

- Executed orders were visually hidden under older pending rows because the Orders tab mixed working and historical orders.
- Legacy rows with `strike=0` and `remainingQuantity=0` could render as active `PENDING` orders.
- Unknown or rejected backend statuses could silently map to frontend `PENDING`.
- Order mutation refresh relied only on normal invalidation timing; the terminal now upserts the returned order and forces active refetches.
- The right rail now owns its scroll area, so activity content is reachable on shorter screens.
- Local API port config was inconsistent; frontend defaults to `http://localhost:8080/api`, so backend local defaults now use `PORT=8080`.
- The user-facing terminal exposes only Orders and Positions as activity views.

Remaining backend hardening before production:

- Wrap order creation, wallet settlement, internal execution ledger creation, and order quantity/status update in a MongoDB transaction.
- Add reconciliation job to flag orders where `status=open` and `remainingQuantity=0`, `strike<=0`, or internal execution totals disagree with order completed quantities.
- Add idempotency on every order submit using `clientOrderId` generated by the frontend.
- Add structured audit events for each order lifecycle transition.
- Add API contract tests that compare `POST /orders`, `GET /orders`, `GET /positions/open`, and wallet balances after each execution scenario.

## Acceptance Checks

- A MARKET BUY immediately shows toast, updated order status, position count, and lower available balance.
- A marketable LIMIT BUY behaves the same as MARKET but keeps `type=LIMIT`.
- A non-marketable LIMIT BUY appears in Orders as cancellable.
- Executed orders never remain visible as cancellable working orders.
- Rows with invalid strike or zero remaining quantity are never cancellable.
- Wallet and positions refresh without waiting for the next poll interval.
